# Метод Фейнмана

Пошаговый разбор механики. Не «что делает» — а «как работает внутри».

## Когда использовать

- «Разбери по шагам» / «как это работает внутри»
- «Объясни механизм» / «почему так»
- Code review, собеседование, отладка
- Пользователь хочет ПОНЯТЬ, не просто знать

## 4 шага Фейнмана

### Шаг 1: Суть в 1 предложении

Что эта вещь делает. Без деталей. Только цель.

```
«Эта функция берёт X, превращает в Y, чтобы Z мог использовать.»
```

### Шаг 2: Разбор на компоненты

Из каких частей состоит. Каждая часть — 1 строка: что делает + зачем.

```
Компонент A → [что делает] → [зачем нужен]
Компонент B → [что делает] → [зачем нужен]
```

### Шаг 3: Пошаговый механизм

Конкретная последовательность. Что за чем происходит.

```
1. Вызывается функция X с аргументами Y
2. Проверяется условие Z
3. Если true → происходит A
4. Если false → происходит B
5. Результат возвращается вызывающему
```

### Шаг 4: Проверочный пример

Конкретные входные данные → конкретный результат. Не абстракция.

```
Вход: { users: [{name: "Аня"}, {name: "Боря"}], filter: "А" }
Шаг 1: filterByName ищет "А" в каждом имени
Шаг 2: "Аня" содержит "А" → оставляем
Шаг 3: "Боря" не содержит "А" → убираем
Выход: [{name: "Аня"}]
```

## Шаблоны промптов

### Шаблон 1: Разбери функцию

```
Объясни эту функцию по методу Фейнмана:

1. Суть в 1 предложении — что она делает
2. Из каких компонентов состоит
3. Пошагово: что происходит от вызова до return
4. Конкретный пример с входными данными и результатом

[вставь код]
```

### Шаблон 2: Объясни архитектуру

```
Разбери архитектуру [системы/файла/модуля]:

1. Главная цель (зачем существует)
2. Слои/компоненты (из чего состоит)
3. Поток данных (откуда → куда → как трансформируется)
4. Зависимости (что от чего зависит)
5. Что произойдёт если убрать/сломать компонент X

[вставь код или опиши систему]
```

### Шаблон 3: Почему это работает (или не работает)

```
Объясни пошагово почему [код/система] работает (или падает):

1. Что происходит при запуске (шаг за шагом)
2. Где именно возникает проблема (или почему всё ок)
3. Какие данные на каждом этапе
4. Что именно идёт не так (или почему всё правильно)

[вставь код/ошибку]
```

## Пример: разбор React useEffect

### Шаг 1: Суть

> `useEffect` запускает код после того как компонент отрисовался на экране.

### Шаг 2: Компоненты

```
useEffect(callback, deps)
  callback     → код который выполнится (после рендера)
  deps (массив)→ при изменении каких значений перезапустить callback
  [] (пустой)  → запустить 1 раз при mount
  (нет)        → запускать при КАЖДОМ рендере
```

### Шаг 3: Пошагово

```
1. Компонент рендерится (JSX → HTML в DOM)
2. React видит useEffect
3. Проверяет deps:
   - deps = [] → callback выполнится 1 раз
   - deps = [x] → если x изменился → callback
   - deps нет  → callback при каждом рендере
4. Callback выполняется (fetch, subscribe, DOM manipulation)
5. При unmount → cleanup функция (если есть return в callback)
```

### Шаг 4: Пример

```jsx
useEffect(() => {
  console.log('count:', count)
  return () => console.log('cleanup')
}, [count])
```

```
Render 1: count = 0
  → useEffect видит deps [count], первый раз → запустит callback
  → console.log('count:', 0)

Render 2: count = 1 (кликнули кнопку)
  → deps изменились (0 → 1) → cleanup + callback
  → console.log('cleanup')
  → console.log('count:', 1)

Render 3: count = 1 (другой стейт изменился, count тот же)
  → deps НЕ изменились → ничего не происходит
```

## Чего избегать

- ❌ Пересказ кода без объяснения ЗАЧЕМ
- ❌ Пропуск шага 4 (пример) — без него объяснение абстрактное
- ❌ «Ну ты понял» — проверь что понял
- ❌ Смешивать уровни (ELI5 + глубокие детали в одном абзаце)

# Подготовка проекта к публичному релизу

Когда проект готов к открытию — структурируй его для внешнего использования.

## Когда использовать

Пользователь говорит:
- «сделай публичным»
- «подготовить к публикации»
- «открыть проект»
- «сделать install скрипт»
- «контрибьюторы»
- «подготовить к релизу»

## Алгоритм действий

### 1. Проверить базовое

- [ ] Есть ли `README.md` с описанием, примерами, установкой?
- [ ] Есть ли `LICENSE` (рекомендуется MIT)?
- [ ] Есть ли `.gitignore`?

### 2. Создать установочные скрипты

**install.sh (Linux/macOS):**
- Копирует основной конфиг в $HOME
- Создаёт необходимые директории
- Цветной вывод, проверка ошибок
- `set -euo pipefail`

**install.ps1 (Windows):**
- Аналог install.sh для PowerShell
- `$env:USERPROFILE` вместо `$HOME`
- `Test-Path` для проверок

### 3. Создать структуру для контрибьюторов

```
.github/
├── CONTRIBUTING.md          # Гайд для контрибьюторов
└── ISSUE_TEMPLATE/
    ├── feature.md           # Предложить фичу
    ├── bug-report.md        # Баг-репорт
    └── documentation.md     # Улучшение документации
```

### 4. Добавить доказательства

- Скриншоты «до» и «после» в `test-results/`
- Сравнительная таблица метрик
- Примеры использования

### 5. Обновить README

Добавить секции:
- Автоматическая установка (curl | bash / irm | iex)
- Контрибьюторы со ссылками на issue templates
- Тесты/результаты
- Лицензия

### 6. Открыть начальные issues

- 🧩 Предложите фичи (enhancement, good first issue)
- 🐛 Баг-репорты (bug)
- 📝 Документация (documentation)

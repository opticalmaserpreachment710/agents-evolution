# TASK Format

## Суть

`TASK.md` нужен как командная доска по одной независимой зоне.

Одна папка `.spec/<slug>/` = одна независимая зона работ.

## Обязательная структура

1. Заголовок `# TASK: ...`
2. Легенда статусов:
   - `[ ]` не сделано
   - `[~]` в работе
   - `[x]` сделано
3. Короткий блок `Как использовать`
4. Блок `Связанные документы`
5. Нумерованные верхние задачи
6. У каждой верхней задачи:
   - статус
   - название
   - `Ответственные`
   - `Зона`
   - `Что считаем результатом`
   - `Подтаски`
   - `Шаги выполнения`
7. У каждой подзадачи:
   - свой статус
   - строка трассировки `_Requirements: ..._`
8. В конце допустим `## Примечания`

## Ответственные

У каждой верхней задачи обязательно должен быть свой блок:

- `owner: ____`

`owner` может быть:

- человеком
- AI-моделью
- названием конкретного исполнителя или команды, если так принято
- списком из нескольких конкретных исполнителей, если задачу делают вместе

## Правила

1. Верхняя задача = независимый блок, который можно отдать одному исполнителю.
2. У каждой верхней задачи должен быть явный `owner`.
3. `owner` может содержать одного исполнителя или несколько исполнителей.
4. Все подтаски и шаги выполнения внутри этой задачи по умолчанию относятся к тому же `owner`.
5. Количество исполнителей не обязано совпадать с количеством задач.
6. Распределение задач задаёт владелец плана: можно делить ровно, неровно или отдавать несколько задач одному исполнителю.
7. Если владелец плана явно просит split по нескольким исполнителям или агентам, при выполнении нужно поднимать отдельные независимые агенты под соответствующие верхние задачи, если инструменты это позволяют.
8. Подтаски = измеримый результат.
9. Шаги выполнения = как именно делать.
10. Трассировка ставится у каждой подзадачи, а не только у верхней задачи.
11. Если задача опциональна для быстрого MVP, можно помечать её `*`.

## Пример подпункта

```md
## 1. [ ] Telegram Web: Надёжность поиска
Ответственные:
- `owner: ____`

Зона:
- поиск групп
- поиск лички

Что считаем результатом:
- живые чаты не теряются после одного UI-сбоя

Подтаски:
- 1.1. [ ] Собрать карту всех точек `not found`
  _Requirements: 1.1, 2.1_

Шаги выполнения:
- 1.2. [ ] Найти текущие точки `search_result_not_found`
  _Requirements: 1.1_
```

# Spec Workflow

## Фаза 0: Сначала понять проект

Эта фаза нужна, когда пользователь ещё не просит делать formal spec, а хочет:

- понять, что можно улучшить
- понять, что шумит
- понять, что опасно
- понять, что чинить первым

В этой фазе:

1. Не создавай `.spec/` автоматически.
2. Сначала разложи проект на независимые зоны.
3. Для каждой зоны покажи:
   - проблема
   - почему важно
   - сейчас -> как лучше
   - что это даст
4. Если уместно, предложи будущие улучшения.

## Фаза 1: Потом формализовать

Переход в spec-фазу происходит только если пользователь явно просит:

- сделать spec
- оформить task/requirements/design
- разложить в `.spec/`

В этой фазе:

1. Создавай `.spec/<slug>/`
2. Делай:
   - `TASK.md`
   - `Requirements.md`
   - `Design.md`
3. В `TASK.md`:
   - каждая верхняя задача имеет свой `owner`
   - подзадачи наследуют этого `owner`
   - у каждой подзадачи есть `_Requirements: ..._`

## Фаза 2: Сборка и проверка после split

Если работа выполняется несколькими исполнителями или агентами, должен быть отдельный ведущий-проверяющий.

Важно:

1. Его не обязательно фиксировать в `TASK.md`.
2. Главное, что он существует как роль.
3. Он не закрывает spec заранее.
4. Он ждёт, пока исполнители закончат свои верхние задачи.

После этого ведущий-проверяющий:

1. собирает результаты всех агентов
2. смотрит, не конфликтуют ли изменения
3. интегрирует всё вместе
4. обновляет один общий `TASK.md`
5. прогоняет финальную проверку
6. закрывает spec по факту

## Простая модель

Сейчас -> потом

- сначала понять, что улучшать
- потом формализовать это в spec

Не наоборот.

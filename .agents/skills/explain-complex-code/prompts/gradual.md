# Gradual Disclosure — От простого к сложному

Постепенное раскрытие деталей. Не вываливать всё сразу. Дать контроль над глубиной.

## Когда использовать

- Большой файл (200+ строк) или сложная архитектура
- Пользователь просит «объясни постепенно» / «от простого к сложному»
- Неясен уровень подготовки — начать с базового, углублять по запросу
- Onboarding в новый проект/кодовую базу

## Принцип

```
Уровень 1: Суть (2-3 предложения)      ← всегда показывать
    ↓ «хочу глубже»
Уровень 2: Структура (архитектура)      ← если спросил
    ↓ «что тут делает X»
Уровень 3: Механизм (как работает)      ← если спросил
    ↓ «разбери функцию Y»
Уровень 4: Детали (edge cases, hacks)   ← если спросил
```

**Никогда не показывай уровень N пока не спросят.**

## Уровни

### Уровень 1: Суть

Цель: человек понимает ЧТО перед ним и ЗАЧЕМ это существует.

```
Это [тип вещи]. Делает [главное действие]. Нужно чтобы [цель].

Ключевые компоненты: [A], [B], [C].

[A] — коротко что делает
[B] — коротко что делает
[C] — коротко что делает
```

После Уровня 1 — **остановись**. Спроси:
> Хотите разобрать структуру подробнее или какой-то компонент глубже?

### Уровень 2: Структура

Цель: человек видит как компоненты связаны.

```
┌─────────────────────────────────┐
│         main() / entry          │  ← точка входа
│    ↓            ↓          ↓    │
│  [A]          [B]         [C]   │  ← компоненты
│    ↓            ↓          ↓    │
│  helpers      utils      config  │  ← поддержка
└─────────────────────────────────┘

[A] → вызывает [X] и [Y]
[B] → зависит от [A]
[C] → независимый, логгирует
```

После Уровня 2 — **остановись**. Спроси:
> Какой компонент разобрать подробнее?

### Уровень 3: Механизм

Цель: человек понимает как работает конкретный компонент.

Используй `feynman.md` — 4 шага Фейнмана для выбранного компонента.

### Уровень 4: Детали

Цель: edge cases, хаки, подводные камни, оптимизации.

- Что сломается если X
- Почему написано именно так (а не «очевидным» способом)
- Скрытые зависимости
- Производительность, race conditions, memory leaks

## Шаблоны промптов

### Шаблон 1: Объясни файл постепенно

```
Объясни этот файл постепенно. Начни с самого простого уровня — 
что это вообще такое и зачем нужно. Не переходи к деталям пока я не спрошу.

После каждого уровня спрашивай что хочу разобрать дальше.

[вставь код]
```

### Шаблон 2: Объясни архитектуру проекта

```
Объясни архитектуру этого проекта. Начни с общей картины — 
какие части есть и как связаны. Потом будем углубляться по запросу.

Сначала:
- Какие компоненты/модули/файлы
- Как они связаны (кто кого вызывает)
- Где точка входа

Не объясняй внутренности пока не спрошу.

[вставь структуру проекта или опиши]
```

### Шаблон 3: Onboarding — как тут всё устроено

```
Я новый в проекте. Объясни как тут всё устроено — от общего к частному.

1. Что за проект (тип, цель, стек)
2. Какие основные директории/модули
3. Как запрос идёт от входа до выхода
4. Где лежит бизнес-логика

Не уходи в детали реализации. Мне нужна карта, не территория.
```

## Пример: постепенный разбор React компонента

### Уровень 1: Суть

> Это компонент `ProductCard`. Показывает товар в каталоге: картинка, название, цена, кнопка «в корзину». Зависит от `useCart` (корзина) и `useRouter` (навигация).

*Хочешь разобрать структуру или какой-то кусок подробнее?*

### Уровень 2: Структура (если спросил)

```
ProductCard
  ├── useState(selectedImage)    ← какая картинка показывается
  ├── useCart()                  ← добавить в корзину
  ├── useRouter()                ← переход на страницу товара
  ├── handleAddToCart()          ← логика кнопки
  └── JSX:
       ├── ImageGallery          ← миниатюры + главная
       ├── ProductInfo           ← название, цена, рейтинг
       └── AddToCartButton       ← кнопка с disabled состоянием
```

*Что разобрать подробнее — хук useCart, handleAddToCart, или JSX?*

### Уровень 3: Механизм (если спросил про handleAddToCart)

> **Шаг 1:** Проверяет `if (!inStock)` — нет товара → exit
> **Шаг 2:** Вызывает `cart.addItem({ id, qty: 1 })`
> **Шаг 3:** `setIsAdding(true)` — включает анимацию кнопки
> **Шаг 4:** `setTimeout(1000)` — через секунду выключает анимацию
> **Шаг 5:** Показывает тост «Добавлено в корзину»

### Уровень 4: Детали (если спросил «почему setTimeout»)

> `setTimeout` здесь — хак чтобы анимация успела проиграться. Без него `setIsAdding(false)` выполнится в том же тике и React отрисует сразу конечное состояние — пользователь не увидит «добавляется...». `setTimeout(..., 1000)` выносит обновление в следующий макротаск, давая CSS-анимации 1 секунду.

## Правила

1. **Всегда останавливайся после уровня.** Не показывай следующий без запроса.
2. **Всегда давай выбор.** «Что хочешь разобрать: A, B, или C?»
3. **Уровень 1 — максимум 5 предложений.** Если больше — ты не умеешь сокращать.
4. **Не предполагаешь что пользователь знает предыдущие уровни.** Кратко напоминай контекст.
5. **Если пользователь перескочил** («объясни сразу как работает debounce») — отвечай на его уровень, не тяни через все ступени.

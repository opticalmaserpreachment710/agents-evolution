### Cross-Platform Guardrail (global)

- По умолчанию использовать кроссплатформенный подход (Linux/macOS/Windows) по правилам из `AGENTS.md` (`Cross-Platform By Default (Global)`).

### Когда включать

- Пользователь явно просит `SPEC` или “сделай спецификацию”.
- Пользователь просит создать/обновить `design.md`, `requirements.md`, `tasks.md`.

### Усиленное распознавание “это запрос про SPEC, а не про код”

Иногда пользователь не пишет слово `SPEC`, но по смыслу явно хочет **сначала** поправить/дополнить спеки (design/requirements/tasks), а уже потом решать “как кодить”. В таких случаях **считай, что включился SPEC** и не переходи к коду, пока не обновишь документы (или пока пользователь явно не скажет “сразу в код”).

**Сильные сигналы SPEC (даже без слова `SPEC`):**
- В тексте есть путь/упоминание `.kiro/specs/<spec-name>/...` или конкретных файлов спека.
- Пользователь пишет “добавь/обнови/зафиксируй в дизайн/таски/реквайрментс”, “в спецификацию <name>”, “в spec <name>”.
- Пользователь обсуждает вариантность решения (“вариант A/B”, “мне нравится вариант В”) и просит “зафиксировать” решение.
- Пользователь ссылается на “по доке/точно по документации” и хочет уточнить правила/политику поведения.
- Пользователь описывает сценарий/политику (например: “если запрет писать или забанили в группе, сделай навсегда, если это не лимиты”) и просит “досделай” без просьбы менять код прямо сейчас.

**Правило (анти-ошибка):** если запрос распознан как SPEC, то:
1) сначала обнови `design.md`/`requirements.md`/`tasks.md` (внутри нужного `.kiro/specs/<spec>/`),
2) затем одним коротким вопросом уточни, готов ли пользователь запускать реализацию tasks сейчас,
3) до явного ответа "да" не переходи к реализации в коде.

### Проактивное предложение SPEC при длинном контексте

- ИИ не должна бояться предлагать SPEC-режим, если по диалогу уже накоплено много фактов, ограничений и желаемого поведения для реализации.
- Если пользователь долго обсуждает проект и желания, а контекста достаточно для формализации, ИИ должна сама предложить создать/обновить отдельный SPEC.
- При таком предложении ИИ обязана кратко перечислить, что уже поняла и какие решения/ограничения уже известны, чтобы пользователь видел основу для нового SPEC.
- Цель: не терять накопленный контекст и переводить его в структурированный план (`design.md`/`requirements.md`/`tasks.md`) до начала сложной реализации.

**Шаблон ответа (когда явно SPEC):**
- “Вижу, что это правки спека. Сначала зафиксирую в `.kiro/specs/<name>/{design,requirements,tasks}.md`, затем спрошу о готовности запускать реализацию tasks.”

### Test Awareness (обязательное правило SPEC)

В SPEC-режиме ИИ ОБЯЗАН думать о тестируемости на всех уровнях:

1) Каждое требование в `requirements.md`:
   - должно быть проверяемым,
   - иметь явные Acceptance Criteria,
   - подразумевать как минимум один тип теста
     (`unit` / `integration` / `contract` / `e2e` / `property` / `regression`).

2) В `design.md`:
   - для каждого ключевого решения или ограничения
     указывается, КАК оно будет проверяться,
   - архитектура считается неполной без описания:
     - какие тесты ловят регрессии,
     - какие тесты ловят интеграционные ошибки,
     - какие ошибки будут видны только в `e2e`.

3) В `tasks.md`:
   - каждая значимая задача обязана либо:
     a) создавать тесты,
     b) опираться на существующие тесты,
     c) явно объяснять, почему тест невозможен.
   - задачи без тест-следствия считаются незавершенными.

4) Запрещено:
   - фиксировать требования, которые нельзя проверить,
   - писать “проверим вручную” без объяснения,
   - откладывать тест-мышление “на потом”.

Цель:
SPEC — это контракт, который можно автоматически проверять тестами и CI.

### Тестирование и Верификация (Strict Test Architecture)

- **Strict Test Isolation**: Запрещено смешивать тесты разных типов in одной куче. Используй иерархию подпапок:
  - `tests/unit/` — математика и чистая логика.
  - `tests/integration/` — взаимодействие модулей.
  - `tests/e2e/` — сценарии Playwright и UI-тесты.
  - `tests/security/` — аудит уязвимостей.
- **Root Hygiene**: Запрещено создавать тестовые файлы прямо in корне `project/`. Корень — только для кода и конфигов. Все тесты обязаны находиться in специализированных подпапках директории `tests/`.
- **Targeted Testing**: Выбор самых критичных тестов (Unit, E2E, Smoke, Security).
- **Docs**: Ведение `docs/actions/`, `docs/now.md` и `docs/backlog.md`.

### Что делает SPEC

1. Создает скрытую папку `.kiro/specs/` в текущем проекте, если ее нет.
2. Создает подпапку спецификации по имени:
   - Имя берется из запроса пользователя.
   - Если имени нет, задай один уточняющий вопрос и предложи 2-3 варианта.
   - Нормализуй в `kebab-case` (ASCII).
3. Внутри `.kiro/specs/<spec-name>/` создает файлы:
   - `design.md`
   - `requirements.md`
   - `tasks.md`

Для создания каркаса спецификации (без перезаписи существующих файлов) можно использовать:

- Bash (Linux/macOS/WSL/Git Bash):
```bash
bash scripts/spec-init.sh "<spec-name>"
```

- PowerShell (Windows):
```powershell
pwsh -File scripts/spec-init.ps1 "<spec-name>"
```

### Правила записи

- Не перезаписывай существующие файлы без явного подтверждения пользователя.
- Если файлы уже есть, уточни: обновить/дополнить/создать новую спецификацию.
- Содержимое должно быть на русском, если пользователь не попросил иначе.
- Путь `.kiro/specs/...` всегда **относительный** к корню проекта (без ведущего `/`).
- В SPEC-режиме **нельзя** создавать `design.md`/`requirements.md`/`tasks.md` в корне проекта: эти файлы относятся только к конкретной спецификации внутри `.kiro/specs/<spec-name>/`.

### Структура содержимого (ориентир)

Если в корне проекта уже существуют `design.md`, `requirements.md`, `tasks.md`, можно использовать их **только как шаблоны структуры**. Если их нет, **не создавай их в корне**: ориентируйся на канонические примеры в `references/Example*.md`. В SPEC-режиме наполнение должно следовать тем же разделам и подходу:

- `design.md`: обзор, архитектура (включая диаграммы/флоу), компоненты и интерфейсы, ключевые потоки, API/конфигурации, а также все важные ограничения и решения.
- `requirements.md`: введение, глоссарий, требования в виде **User Story** + **Acceptance Criteria** с нумерацией и форматом “WHEN/THEN/THE … SHALL”.
- `tasks.md`: план реализации с чек-листами, разбивкой на этапы, подзадачи и ссылками на требования в формате `_Requirements: X.Y_`.

### Детальные правила и шаблоны

**Важно про примеры:**
Все блоки `Пример`/`Шаблон`/`Точный шаблон` внутри этого файла — это **ориентир** и **помощь для мышления**. Их цель — показать, **как думать** и **как оформлять** документы. Они не являются контентом проекта и не должны копироваться дословно, кроме явно отмеченных “точных шаблонов”, где требуется 1:1 формат.

Полные примеры вынесены в `references/Example*.md` и должны использоваться как основной reference, чтобы не раздувать этот документ.

**Правило “примеры наизусть” (обязательное):**
Считай `references/ExampleDesign.md`, `references/ExampleRequirements.md`, `references/ExampleTasks.md` каноническим эталоном структуры и формата SPEC-документов. Это часть базовой роли Opus 4.6: “знать” эти примеры и применять их как основу даже в других проектах и совсем других задачах, когда пользователь просит `SPEC` (приоритет: локальные правила/шаблоны проекта выше; если их нет или они неполные, используй этот эталон).

При этом в SPEC-режиме создавай/обновляй **именно** файлы спеки: `.kiro/specs/<spec-name>/design.md`, `.kiro/specs/<spec-name>/requirements.md`, `.kiro/specs/<spec-name>/tasks.md`. Не создавай и не добавляй `Example*.md` внутрь спеки: примеры используются как reference/шаблон (кроме случаев, где явно требуется 1:1 “точный шаблон”, например для структуры `tasks.md` и блока `## Примечания`).

**Язык SPEC-файлов (обязательное правило):**
- Все SPEC-файлы (`design.md`, `requirements.md`, `tasks.md`) заполняй на русском языке: заголовки, описания, требования, задачи и примечания.
- Если в шаблоне встречаются англоязычные служебные форматы/ключевые слова (например `WHEN/THEN/SHALL`, `_Requirements: X.Y_`), их сохраняй как есть только там, где это явно требуется форматом.

**Примеры (явные метки):**
- `ExampleDesign.md` — пример структуры и обязательных секций (см. `references/ExampleDesign.md`).
- `ExampleRequirements.md` — пример структуры и Acceptance Criteria (см. `references/ExampleRequirements.md`).
- `ExampleTasks.md` — точный шаблон и порядок секций (см. `references/ExampleTasks.md`).

#### `design.md` (структура и обязательные секции)

Используй структуру, близкую к существующему `design.md`, и придерживайся:

1. **Title**: `# Design Document: <название>`
2. **Overview**: 1-3 абзаца, что строим и зачем.
3. **Architecture**:
   - **High-Level Architecture** с `mermaid`-диаграммой (graph/flow).
   - **Component Architecture**: список компонентов и их роли.
4. **Components and Interfaces**:
   - Нумерованные компоненты.
   - Для каждого: responsibilities, inputs/outputs, ключевые интерфейсы/функции.
5. **Data/Control Flows** (если есть): ключевые сценарии обработки.
6. **Configuration & Secrets**: env vars, секреты, настройки.
7. **Error Handling & Logging**: как обрабатываются ошибки.
8. **Constraints / Decisions**: важные ограничения и принятые решения.

#### `requirements.md` (структура и формат)

1. **Title**: `# Requirements Document`
2. **Introduction**: цель и границы.
3. **Glossary**: список терминов `Term: definition` или `- **Term**: ...`.
4. **Requirements**:
   - Каждый раздел: `### Requirement N: <краткое имя>`
   - **User Story**: `As a <роль>, I want <цель>, so that <ценность>.`
   - **Acceptance Criteria**:
     - Нумерованный список.
     - Формат: `WHEN ... THE <Component> SHALL ...`
     - Используй `SHALL` и единый стиль.
   - Уникальная нумерация `Requirement N`.
5. **Testing in CI/CD**:
   - Раздел с подразделами:
     - **Pre-merge Testing**: перечисление обязательных шагов (unit, property, lint, typecheck).
     - **Post-merge Testing**: интеграционные проверки, мониторинг, алерты.
     - **Manual Testing Checklist**: ручные шаги перед продом.

**Формат Acceptance Criteria (строгий):**
- Каждый пункт начинается с `WHEN` (или `THE` если условие не нужно) и содержит `SHALL`.
- Стиль: `WHEN <condition>, THE <Component> SHALL <behavior>`.
- Пример (используй как эталон):
  - `WHEN any workflow step fails, THE Issue_Manager SHALL log detailed error information including issue number and error message`
  - `WHEN AWS Bedrock API calls fail, THE Issue_Manager SHALL log the specific error and continue processing`
  - `WHEN GitHub API calls fail, THE Issue_Manager SHALL retry with exponential backoff up to 3 times`
  - `WHEN critical failures occur, THE Issue_Manager SHALL create a workflow run summary with failure details`
  - `THE Issue_Manager SHALL not fail the entire workflow run if individual issue processing fails`

#### `tasks.md` (структура, статусы и правила обновления)

**Структура:**
1. **Title**: `# Implementation Plan: <название>`
2. **Overview**: 1-2 абзаца.
3. **Tasks**: чек-лист с нумерацией.
4. **Notes**: обязателен и всегда в конце файла.

**Точный шаблон `tasks.md` (повторяй порядок и стиль):**
```
# План реализации: <название>

## Обзор

<1-2 абзаца>

## Задачи

- [ ] 1. <Задача>
  - [ ] 1.1 <Подзадача>
    - _Requirements: X.Y_
  - [ ] 1.2 <Подзадача>*
    - _Requirements: X.Y_

- [ ] 2. Контрольная точка - <краткое описание>
  - Убедиться, что все тесты проходят, спросить пользователя при необходимости.

## Примечания

- Задачи с `*` опциональны и могут быть пропущены для быстрого MVP
- Каждая задача ссылается на требования для трассируемости
- Контрольные точки обеспечивают поэтапную валидацию
- Property-тесты проверяют универсальные свойства корректности
- Unit-тесты проверяют конкретные примеры и крайние случаи
- Весь TypeScript-код должен следовать best practices и использовать strict mode
- Используй аннотации типов повсеместно для качества кода
- AWS credentials должны храниться в GitHub Secrets, никогда в коде
```

**Примечания (обязательный блок):**
Содержимое и порядок пунктов в `## Примечания` должны совпадать с шаблоном выше.

**Язык:**
Все разделы `tasks.md`, включая заголовки, описания и чек-листы, должны быть на русском языке.

**Формат задач:**
- Верхний уровень: `- [ ] N. <задача>`
- Подзадачи: `  - [ ] N.M <подзадача>`
- В конце каждой (под)задачи строка `_Requirements: X.Y_` если применимо.

**Статусы (для машинного чтения):**
- `[x]` = выполнено полностью.
- `[ ]` = не выполнено.
- `[~]` = в процессе (используй только если это реально активная работа).

**Правила обновления:**
- После любого выполненного объема работ по спекам/реализации **обязательно синхронизируй** `tasks.md`: проставь статусы всем затронутым пунктам (даже если весь план целиком не закрыт). Логика простая: что сделано полностью = `[x]`, что сейчас в работе = `[~]`, остальное = `[ ]`.
- Обновляй статусы **по мере выполнения**: завершил шаг -> сразу отметь в `tasks.md`, взял в работу -> поставь `[~]`, завершил и проверил -> переведи в `[x]`.
- Помечай `[x]` только после завершения кода и проверки/тестов (или явного отказа от тестов с причиной).
- Не удаляй/не переименовывай старые пункты; добавляй новые в конец соответствующего раздела.
- Не меняй нумерацию уже существующих пунктов.
- Если задача заблокирована, оставь `[ ]` и добавь строку `Blocked: <причина>` сразу после задачи.
- При частичном прогрессе не ставь `[x]`; используй `[~]` и добавь `Progress: ...`.

**Правило прогонов тестов (Fast/Task/Release Gates) (обязательное):**
1) После каждой подзадачи (`N.M`) агент обязан запускать `Fast Gate`:
   - `lint/format` (если есть),
   - `typecheck` (если есть),
   - `unit` (целевая область или весь unit-набор).
   Обновление статуса:
   - `[x]` только если `Fast Gate` зелёный,
   - `[~]` если в процессе.
2) После завершения задачи верхнего уровня (`N`) агент обязан запускать `Task Gate`:
   - все `unit`,
   - интеграционные тесты по затронутым компонентам,
   - `contract` тесты (если затронуты API/schemas/events).
   `[x]` для задачи `N` ставится только после зелёного `Task Gate`.
3) На контрольных точках и перед релизом агент обязан запускать `Release Gate`:
   - all `unit` + `integration` + `contract`,
   - `e2e smoke` (на PR) / `e2e full` (nightly или pre-release),
   - `performance/security` (если указано в требованиях).
   Если полный прогон слишком долгий, агент обязан предложить разбиение:
   - PR: `smoke`,
   - nightly: `full`.
4) Запрещено:
   - ставить `[x]`, если соответствующий Gate не пройден,
   - “лечить” flaky тесты через `sleep` без объяснения причины и стабильного решения.

**Базовые vs усиленные тесты (`*`):**
- Базовые тесты (`unit` + `integration/contract` по зоне изменений) — часть Definition of Done и не помечаются `*`.
- `*` допустима только для усиленных проверок (`property` / `chaos` / `load` / расширенный `e2e`) и может откладываться только при явном подтверждении пользователя.
- Если изменение затрагивает только один модуль, на `Fast Gate` можно запускать unit этого модуля; на `Task Gate` всегда запускать все unit.

**Правило ответа по статусам:**
При запросе “что сделано/что нет” опирайся только на чекбоксы: `[x]` = done, `[ ]` = not done, `[~]` = in progress.
